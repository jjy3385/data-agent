# AdventureWorks2022 기반 MVP E2E 시나리오 후보

> 이 문서는 AdventureWorks2022의 실제 데이터를 기반으로 검토한 MVP E2E 시나리오 후보와 대표 질문 선정 근거를 설명한다. 왜 현재 공급 리스크 질문을 선택했는지 확인할 때 읽는다.

최종 범위는 [MVP 범위](scope.md), 완료 판정은 [MVP 완료 기준](acceptance-criteria.md)을 따른다.

사용자는 자연스러운 업무 언어로 질문하고, LLM은 Runtime Intent를 구조화한 뒤
ACL로 허용된 Entity, Dimension, Metric, 기간 정책과 승인 Join을 조합해 Query
Plan과 MSSQL을 생성한다. Agent는 Raw Schema의 컬럼 이름이나 일반 업무 상식에서
업무 의미를 임의로 추론하지 않으며, 필요한 Metadata가 없으면 Fail Closed 처리한다.

## 1. 후보 비교 요약

5점에 가까울수록 해당 평가 항목에 적합하다.

| 후보 | 데이터 적합성 | 파편화 해결 | Agentic 적합성 | 4주 적합성 | 종합 판단 |
|---|---:|---:|---:|---:|---|
| 재고 부족 및 공급 리스크 | 4 | 5 | 5 | 4 | 1순위 |
| 공급업체 품질 및 사업 영향 | 4 | 5 | 5 | 3 | 2순위 |
| 매출 이상 징후 드릴다운 | 5 | 3 | 3 | 5 | 안전한 대안 |
| 생산 스크랩 및 사업 영향 | 4 | 4 | 4 | 3 | 확장 후보 |
| 영업 목표 미달 원인 분석 | 4 | 4 | 4 | 4 | ACL 강조 후보 |

## 2. 후보 1: 재고 부족 및 공급 리스크 통합 분석

### 2.1 질문 예시

> 최근 판매와 재고·공급 상황을 함께 봤을 때 우선 확인해야 할 품목을 알려줘.

사용자는 90일, 안전재고 공식, 재고 집계 방식이나 리드타임 컬럼을 알 필요가
없다. LLM은 질문에 맞는 승인 Metadata를 조합해 Query Plan을 만들고, 백엔드
검증을 통과한 Plan을 기반으로 MSSQL을 생성한다.

### 2.2 필요한 승인 Metadata

- “최근”의 기본 기간과 기준일
- 판매수요의 대상 날짜, 수량 컬럼과 집계 Grain
- 품목별 재고 합계와 안전재고 미달 공식
- 공급업체 수와 리드타임의 단위·집계 방식
- 구매 거부 이력의 업무 의미와 계산식
- 검토 우선순위 규칙과 TOP N
- 허용된 Physical FK 또는 Virtual FK
- 수요예측·품절예측·자동발주로 해석하지 않는다는 사용 제한

### 2.3 연결하는 업무 데이터

- `Production.Product`: 품목, 안전재고, 재주문점
- `Production.ProductInventory`: 창고별 재고
- `Production.Location`: 창고 및 생산 위치
- `Sales.SalesOrderHeader`, `Sales.SalesOrderDetail`: 최근 판매수요
- `Purchasing.ProductVendor`: 품목별 공급업체와 평균 리드타임
- `Purchasing.Vendor`: 공급업체 상태와 신용등급
- `Purchasing.PurchaseOrderHeader`, `Purchasing.PurchaseOrderDetail`: 구매 및 거부 수량

기존 패키지 솔루션에서는 품목 기준정보, 창고 재고, 판매실적, 공급업체 등록,
구매 및 입고 품질 화면을 각각 열고 값을 비교해야 할 가능성이 높다. 이
시나리오는 흩어진 값을 품목이라는 하나의 업무 맥락으로 묶는다.

### 2.4 실제 데이터 예시

다음 수치는 후보 검증을 위해 위와 같은 Metric 정의를 적용한 결과다. 운영
Agent가 컬럼 이름만 보고 자동으로 만들어낸 업무 정의가 아니다.

| 품목 | 재고 | 안전재고 | 최근 90일 수요 | 공급업체 | 최소 평균 리드타임 |
|---|---:|---:|---:|---:|---:|
| HL Mountain Tire | 499 | 500 | 351 | 2 | 19일 |
| Hitch Rack - 4-Bike | 0 | 4 | 349 | 1 | 45일 |
| Half-Finger Gloves, M | 0 | 4 | 280 | 1 | 15일 |
| LL Road Tire | 488 | 500 | 245 | 2 | 19일 |
| Touring Tire | 482 | 500 | 232 | 2 | 19일 |

### 2.5 제한된 Agentic Workflow

1. 요청자의 ACL을 확인하고 허용된 Metadata만 Context에 포함한다.
2. LLM이 Runtime Intent와 Depth 1 Query Plan을 만들고 백엔드가 검증한다.
3. **Depth 1 — 위험 선별:** 검증된 Plan과 승인 Metadata의 안전재고 미달 및 최근 수요를
   결합해 위험 품목 Top N을 찾는다.
4. 첫 번째 조회 결과를 평가해 상세 조사가 필요한 품목만 고른다.
5. **Depth 2 — 근거 조사:** 선택된 품목의 창고별 재고, 공급업체 수,
   리드타임, 구매 거부 이력을 조회한다.
6. 결과를 “긴급 검토”, “모니터링”, “공급처 확인 필요” 같은 검토 우선순위로
   정리하고, 사용한 기준과 데이터 한계를 설명한다.
7. 계획, SQL, Guardrail 결과, 실행 결과와 최종 답변을 Audit Log에 저장한다.

정책 검사와 Guardrail은 Agent Depth에 포함하지 않는다. SQL 실행 오류에 대한
1회 Self-Healing도 업무 탐색 Depth와 별도로 관리하되, 재생성된 SQL은 동일한
Guardrail을 다시 통과해야 한다.

### 2.6 장점

- Sales, Inventory, Purchasing 세 영역의 화면 파편화를 명확히 해결한다.
- 첫 조회 결과에 따라 두 번째 조회 대상을 결정하므로 Agentic 특성이 분명하다.
- 실제 데이터에 검토 대상이 존재해 데모와 자동 E2E 테스트가 안정적이다.
- 읽기 전용 분석과 잘 맞으며 외부 시스템 쓰기 작업이 필요하지 않다.
- ACL, Metadata, Join, Guardrail, XAI, Audit을 한 시나리오에서 검증할 수 있다.

### 2.7 단점과 표현 제한

- 최근 판매량은 수요 예측값이 아니라 과거 실적이다.
- 예약재고, 가용재고, 미입고 발주잔량을 완전하게 계산할 수 없다.
- 테이블별 데이터 시점이 서로 다르다.
- 정확한 품절 예상일이나 자동 발주량을 산정할 근거가 부족하다.

따라서 결과를 “품절 예측” 또는 “발주 지시”라고 표현하지 않고,
**재고·수요·조달 조건을 결합한 검토 우선순위**라고 표현한다.

## 3. 후보 2: 공급업체 품질 문제와 사업 영향 분석

### 3.1 질문 예시

> 최근 구매 품질과 재고·판매 영향을 함께 봤을 때 우선 확인할 공급업체나 품목을 알려줘.

이 질문을 사용하려면 “최근 구매”, “품질”, “영향”의 의미와 구매 거부율 공식,
대체 공급업체 판정 기준을 관리자가 먼저 승인해야 한다.

### 3.2 Agentic Workflow

1. **Depth 1:** 공급업체 또는 공급업체-품목별 구매 거부율과 거부 수량을
   계산해 이상 대상을 선별한다.
2. **Depth 2:** 선별된 품목의 재고, 최근 판매수요, 다른 등록 공급업체와
   리드타임을 조사한다.
3. 품질 문제와 잠재적 사업 영향을 구분해 근거와 함께 반환한다.

### 3.3 실제 데이터 예시

- 구매 거부 수량 합계는 72,700개이고 관련 구매 라인은 563개다.
- `SUPERSALES INC.`의 거부 수량은 6,373개다.
- `Signature Cycles`의 거부 수량은 2,940개이며 집계 거부율은 약 5.12%다.
- `LL Road Tire`는 특정 공급업체 구매에서 거부 수량 1,510개가 있으며,
  재고 488, 안전재고 500, 최근 90일 판매수요 245로 확인된다.

### 3.4 장점

- “문제 발견 → 영향 범위 조사”라는 Agentic 흐름이 명확하다.
- 구매, 품질, 재고, 판매 화면에 흩어진 근거를 연결한다.
- 대체 공급업체 존재 여부까지 확인하면 실무적인 의사결정 보조가 된다.

### 3.5 단점

- 실제 입고 완료 시각이 없어 납기 준수율은 계산할 수 없다.
- 샘플 데이터의 거부 수량 분포가 현실 업무보다 인위적일 수 있다.
- 구매 부품이 완제품에 미치는 영향을 계산하려면 다단계 BOM 탐색이 필요하다.
- BOM 재귀 탐색까지 포함하면 4주 MVP 범위를 넘길 가능성이 높다.

MVP에서는 직접 판매되는 품목 또는 직접 재고 영향까지만 다루고, 다단계 BOM
영향 분석은 후속 단계로 미룬다.

## 4. 후보 3: 매출 이상 징후 드릴다운

### 4.1 질문 예시

> 최근 매출이 눈에 띄게 달라진 제품군과 함께 봐야 할 관련 변화를 알려줘.

비교 기간, 매출 공식, “눈에 띄는 변화”의 임계값과 허용 드릴다운 차원은 승인
Metadata로 정의해야 한다.

### 4.2 흐름

1. **Depth 1:** 월별 매출 변동이 큰 제품군 또는 지역을 선별한다.
2. **Depth 2:** 선별된 대상의 고객 유형, 제품, 할인, 지역 구성을 드릴다운한다.

### 4.3 장점

- 판매 데이터량이 충분해 결과가 안정적으로 나온다.
- 구현 난도가 상대적으로 낮고 4주 내 성공 가능성이 가장 높다.
- 월별 추세에서 제품, 지역, 고객으로 자연스럽게 상세 분석할 수 있다.

### 4.4 단점

- 일반적인 BI 드릴다운과 차별성이 약할 수 있다.
- 발견한 연관성을 매출 감소의 인과관계로 단정하면 안 된다.
- 판매 영역 안에서만 분석하면 업무 화면 파편화 해결 메시지가 약하다.
- 마지막 월 데이터가 완전한 기간인지 별도 검증이 필요하다.

일정 위험이 커졌을 때 1순위 시나리오를 대체할 수 있는 안전한 후보로 본다.

## 5. 후보 4: 생산 스크랩 비용과 사업 영향 분석

### 5.1 질문 예시

> 최근 생산 손실이 큰 제품과 우선 조사해야 할 원인을 알려줘.

“생산 손실”의 대상 기간, 스크랩 비용 공식, 적용 원가와 조사 우선순위는 승인
Metadata로 정의해야 한다.

### 5.2 실제 데이터 예시

- 스크랩 발생 작업지시는 729건, 스크랩 수량 합계는 10,651개다.
- `ML Crankset`의 `Thermoform temperature too low` 사유는 스크랩 73개이며,
  현재 표준원가를 이용한 단순 영향 추정액은 약 8,313이다.
- `Touring Rear Wheel`의 같은 사유는 스크랩 60개이며 단순 영향 추정액은 약
  6,527이다.

### 5.3 장점

- MES, 재고, 판매 화면을 연결하는 제조업 중심 시나리오다.
- 생산 문제 발견 후 사업 영향까지 조사하는 흐름이 Agentic Workflow에 맞는다.

### 5.4 단점

- 스크랩 사유가 작업지시 단위이므로 특정 공정이나 위치가 원인이라고 단정하기
  어렵다.
- 판매 영향은 제품 수준의 간접 추론이다.
- Routing과 원가까지 정확히 해석하려면 Metadata와 검증 SQL이 복잡해진다.

## 6. 후보 5: 영업 목표 미달 담당자와 원인 분석

### 6.1 질문 예시

> 영업목표 달성이 어려운 담당자와 함께 확인해야 할 실적 변화를 알려줘.

목표 적용 기간, 실적 귀속 기준, 달성률 공식과 담당자 정보의 ACL은 승인
Metadata로 정의해야 한다.

### 6.2 연결하는 데이터

- `Sales.SalesPersonQuotaHistory`
- `Sales.SalesPerson`, `Sales.SalesTerritory`
- `HumanResources.Employee`, `Person.Person`
- `Sales.SalesOrderHeader`, `Sales.SalesOrderDetail`
- `Sales.Customer`, `Sales.Store`

### 6.3 장점

- HR, CRM, Sales 화면의 파편화를 보여줄 수 있다.
- 일반 직원과 영업관리자에게 서로 다른 범위를 보여주는 ACL 데모에 적합하다.

### 6.4 단점

- 영업사원이 17명으로 데이터 규모가 작다.
- `QuotaDate`와 목표 적용 기간의 업무 의미를 Metadata로 명확히 정의해야 한다.
- 인사정보와 성과정보를 연결하므로 권한 설계와 마스킹 부담이 커진다.
- 목표 미달 원인을 데이터만으로 단정하기 어렵다.

## 7. 데이터 파편화 해결의 MVP 정의

AdventureWorks2022는 여러 업무 스키마로 나뉘지만 하나의 MSSQL 데이터베이스이며
물리 FK도 비교적 잘 정의되어 있다. 이 데이터셋만으로 여러 독립 시스템을
통합했다고 주장해서는 안 된다.

4주 MVP에서는 목표를 다음과 같이 한정한다.

> 서로 다른 업무 모듈과 화면에 흩어진 데이터를 관리자가 승인한 관계와 업무
> 정의에 따라 하나의 근거 기반 업무 브리핑으로 조합한다.

MVP의 결과물은 모든 데이터를 모은 거대한 통합 화면보다 질문별 **업무 케이스
카드**에 가깝다. 재고 위험 시나리오라면 다음 근거를 한 번에 제공한다.

- 어떤 품목을 우선 검토해야 하는가
- 어떤 규칙으로 위험하다고 판단했는가
- 창고별 재고는 얼마인가
- 최근 판매수요는 얼마인가
- 등록된 공급업체와 리드타임은 어떠한가
- 과거 구매 거부 이력이 있는가
- 어떤 데이터와 SQL을 사용했는가
- 사실, 계산값, 제한된 추론 중 어디에 해당하는가

첫 조회 결과를 보고 추가 증거가 필요한 대상을 선택한다는 점이 단순 통합 조회나
Text-to-SQL과의 핵심 차이다.

실제 다중 시스템, 다중 데이터베이스, 서로 다른 ID 체계의 연결은 Post-MVP로
둔다. 현재 MVP는 단일 MSSQL 안의 교차 모듈 오케스트레이션을 증명한다.

## 8. 4주 MVP 범위 평가

현재 저장소는 데이터베이스와 설계 문서 중심의 초기 상태다. README에 나열된
전체 기능을 범용 제품 수준으로 구현하는 것은 4주 일정에 공격적이다. 하지만
대표 시나리오 하나를 얇은 수직 범위로 고정하면 완료 가능성이 있다.

### 8.1 반드시 포함할 범위

- AdventureWorks2022 단일 MSSQL 연결
- 자연스러운 대표 질문 1개와 제한된 표현 변형
- 등록된 Entity, Dimension, Metric과 승인된 Join의 동적 조합
- Runtime Intent와 Depth 1·2 Query Plan 생성 및 백엔드 검증
- 사전에 승인한 8~10개 테이블과 Join Metadata
- 사전에 정의하고 검증한 기간 정책, 업무 지표, 집계 Grain, 단위와 사용 제한
- `supply_risk_analyst`, `inventory_viewer` 역할의 ACL과 Slack 사용자 매핑
- Depth 1 위험 선별과 Depth 2 근거 조사
- SQL 오류에 대한 최대 1회 Self-Healing
- SELECT-only, 허용 테이블, TOP N, Timeout Guardrail
- Slack 질문 및 응답
- XAI 설명, Audit Log, 오류 신고
- 정상·권한 거부·위험 SQL·실행 오류 E2E 테스트

### 8.2 범위에서 제외할 항목

- 임의의 업무 질문을 처리하는 범용 자율 에이전트
- 71개 전체 테이블의 완전한 업무 의미 자동 분석
- 다단계 BOM 영향 탐색
- 정교한 수요예측, 품절 예상일 및 최적 발주량
- 자동 발주나 외부 업무 시스템 쓰기
- 다중 MSSQL 및 이기종 시스템 통합
- 무제한 계획, 재귀 탐색 또는 반복 Self-Healing
- 완성형 Admin Portal

### 8.3 권장 4주 진행 단위

| 주차 | 목표 |
|---|---|
| 1주 | Entity, Dimension, Metric, Join과 기간 정책 정의, 읽기 전용 연결, ACL Context 구축 |
| 2주 | Runtime Intent, Depth 1 Query Plan·SQL 생성, 검증, Guardrail, Golden Query 테스트 |
| 3주 | Depth 2 근거 조사, XAI, Slack, Audit Log 연결 |
| 4주 | 오류 신고, 실패 경로, E2E 회귀 테스트와 데모 안정화 |

Metadata CRUD나 범용 Schema Collection보다 대표 시나리오의 Golden Query와 실패
경로를 먼저 완성하는 것이 일정 위험을 낮춘다.

## 9. 권장 대표 시나리오

현재 데이터와 프로젝트 방향을 함께 고려한 자연스러운 대표 질문은 다음과 같다.

> 최근 판매와 재고·공급 상황을 함께 봤을 때 우선 확인해야 할 품목을 알려줘.

LLM은 이 질문에서 Runtime Intent를 구조화하고, 관리자가 승인한 기본 기간,
판매수요, 재고, 안전재고 미달, 공급업체, 리드타임과 구매 거부 이력 Metadata를
조합해 Query Plan과 MSSQL을 생성한다. 구체적인 90일 기준이나 컬럼·집계 공식은
사용자 질문에 요구하지 않고 XAI 응답으로 공개한다.

이 시나리오는 다음 이유로 대표 E2E 시나리오에 적합하다.

- 실제 데이터에서 반복 검증 가능한 결과가 나온다.
- Sales, Production, Purchasing 모듈의 데이터 파편화를 연결한다.
- 최대 2-Depth라는 Agent Workflow 제한 안에서 의미 있는 탐색을 보여준다.
- Runtime Intent, Query Plan, Metadata, ACL, 승인된 Join, Guardrail, XAI, Audit Log를 한
  흐름에서 검증할 수 있다.
- 자동 발주나 예측 모델 없이도 실무적인 의사결정 보조 가치를 전달한다.
- 등록된 Metadata 조합 범위와 최대 2-Depth로 범용성을 제한해 4주 MVP 범위를 통제할 수 있다.

구현 전에는 이 시나리오를 기준으로 다음 항목을 확정해야 한다.

- Runtime Intent와 Query Plan의 구조화 출력 계약
- 정확한 지표 공식, 기준일, 기본 기간과 허용 Parameter
- 집계 Grain, 단위와 사용 제한
- 포함할 테이블과 컬럼 Allowlist
- 관리자 승인 Join 및 업무 용어 Metadata
- 역할별 조회 범위와 민감 컬럼 정책
- Depth 1과 Depth 2의 입력·출력 계약
- XAI 응답 형식
- 정상 및 실패 E2E Acceptance Criteria
